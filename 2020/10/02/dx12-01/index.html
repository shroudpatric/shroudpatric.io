DX12 笔记 04 章

Created2020-10-02|Updated2023-11-13|C++
|Post View:

第四章的内容非常重要，必须要理解之后才能进行后面的深入学习，我建议多看几遍，四遍以上吧。
以下是内容总结。

4.1 预备知识

这一章的知识都是预备知识，可能暂时不会立即用到，但是需要熟练掌握。

4.1.1 D3D 12 概述

D3D 和硬件驱动控制 GPU，上层开发者不需要关心硬件的具体规格和硬件控制层面的实现细节。

4.1.2 COM(Component Object Model)

COM 设计模式使我们无须关心底层细节，并且对以后的 API 升级也不用做特别考虑。

4.1.3 纹理格式

不是任意类型的元素都能用于组成纹理，它只能存储 DXGI_FORMAT 枚举类型中描述的特定格式的数据元素。
纹理不仅可以存储图像数据，也可以存储顶点信息，具体格式有一个专门的枚举值来定义。
并且还有无类型（typeless）的纹理。

  • https://learn.microsoft.com/zh-cn/windows/win32/direct3d9/basic-texturing-concepts
  • https://learn.microsoft.com/zh-cn/windows/win32/direct3d11/overviews-direct3d-11-resources-textures-intro
  • https://learn.microsoft.com/zh-cn/windows/win32/direct3d11/overviews-direct3d-11-resources-textures

4.1.4 交换链

back buffer 和 front buffer 相互交换显示给屏幕，形成交换链。
使用两个 buffer 的称为 double buffering，三个 buffer 的称为 triple buffering。

4.1.5 深度缓冲

深度缓冲区（depth buffer）存储特定像素的深度信息[0.0,1.0]。大小和后台缓冲区相同，且一一对应。
深度缓冲技术的原理是计算每个像素的深度值，并执行深度测试（depth test）。
为了确定物体的前后顺序，D3D 采用了 z-buffering 或者叫做 depth-buffering 的技术，并且开启了 depth - buffering 后物体的渲染顺序就无关紧要了。
depth buffer 也是一种纹理，需要对应的数据格式来创建。有以下几种格式。

  • DXGI_FORMAT_D32_FLOAT_S8X24_UNIT : float 32 指定浮点型深度缓冲区，8位无符号整数分配给模板缓冲区，并且映射到 [0,255],剩下24填充对齐无意义。
  • DXGI_FORMAT_D32_FLOAT : 指定一个 32 位浮点型深度缓冲区。
  • DXGI_FORMAT_D24_UNORM_S8_UNIT : 指定一个无符号 24 位深度缓冲区，并将元素映射到 [0,1] 区间。另外 8 为分配给 stencil buffer，映射到 [0,255]。
  • DXGI_FORMAT_D16_UNORM : 指定一个无符号 16 位 深度缓冲区，映射到 [0,1]。

4.1.6 资源与描述符

descriptor 或者 view 描述资源、资源格式、资源用途。
一个资源可以有多个描述符。
descriptor heap 是 descriptor 的集合，
常用描述符

  • CBV (Constant buffer view) 常量缓冲区视图
  • SRV (shader resource view) 着色器资源视图
  • UAV (unordered access view) 无序访问视图
  • Sampler descriptor 采样器描述符
  • RTV (Render Target View) 渲染目标视图
  • DSV (Depth/Stencil View) 深度模板视图

如果以无类型的格式创建了一个资源，又希望该纹理中的元素可以根据需求当做浮点值或者整数值来使用，那么就需要为它创建两个描述符，
一个指定为浮点格式，一个指定为整数格式。

4.1.7 多重采样技术的原理

back buffer 扩大四倍，然后：

  • SSAA: 显示到前面时求四个像素的平均值输出和屏幕大小相同的像素数量。
  • MSAA: 显示到前面时计算一次像素中心的颜色，然后根据 depth/stencil 分享给其他像素。

4.1.9 功能级别

enum D3D_FEATURE_LEVEL
{
D3D_FEATURE_LEVEL_9_1 = 0x9100,
D3D_FEATURE_LEVEL_9_2 = 0x9200,
D3D_FEATURE_LEVEL_9_3 = 0x9300,
D3D_FEATURE_LEVEL_10_0 = 0xa000,
D3D_FEATURE_LEVEL_10_1 = 0xa100,
D3D_FEATURE_LEVEL_11_0 = 0xb000,
D3D_FEATURE_LEVEL_11_1 = 0xb100
} D3D_FEATURE_LEVEL;

应用程序当按照从最新到最旧的级别支持顺序展开检测：

  • 首先检测Direct3D 11是否被支持，
  • 其次检测 Direct3D 10，
  • 最后检测Direct3D 9.3。
    在本书中，我们总是假设需要支持的功能级别为 D3D_FEATURE_LEVEL_11_0 。但是在现实的应用程序中，我们往往需要考虑支持稍旧的硬件，以获得更多的用户。

4.2 CPU 与 GPU 间的交互

同步是一种我们不乐于执行的操作，因为这意味着一种处理器要以空闲状态等待另一种处理器完成某些任务，换句话说，它破坏了两者并行工作的机制。

4.2.1 命令队列和命令列表

我们希望 CPU 和 GPU 同时忙碌，而不是任何一个等待另一个执行完毕。
一个命令分配器可以关联多个命令列表，但是不能同时用它们来记录命令。

4.2.2 CPU 与 GPU 间的同步

强制 CPU 等待，直到 GPU 完成所有命令的处理，达到某个指定的 fence point。这种方式称为 flushing the command queue。
我们几乎可以在任何时间点上刷新命令队列，不一定仅在渲染每一帧时才刷新一次。

4.2.3 资源转换

为了防止 resource hazard（资源冒险），对资源进行资源转换。根据 D3D 给出的转换信心，GPU 就可以采取适当的措施避免资源冒险的发生。

4.2.4 命令与多线程

命令列表、命令分配器并非自由线程对象。

命令队列是线程自由的对象。

深度缓冲区其实就是一种 2D 纹理，它存储着离观察者最近的可是对象的深度信息（如果使用了模板，还会附有模板信息）。
在使用深度模板缓冲区之前，一定要创建相关的深度/模板视图，并将它绑定到渲染流水线上。

4.3.9 设置视口

我们通常会将 3D 场景绘制到与整个屏幕或者整个窗口工作区大小相当的后台缓冲区中。但是有时只是希望把 3D 场景绘制到后台缓冲区的某个
矩形子区域当中，我们把后台缓冲区的这种矩形子区域叫做视口（viewport）。

4.3.10 设置裁剪矩形

我们可以相对于后台缓冲区自定义一个裁剪矩形（scissor rectangle），在此矩形外的像素都将被剔除（即这些图像部分将不会被光栅化
（rasteize）只后台缓冲区）。
